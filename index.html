取り組む課題

　近い将来、下記が大きな問題となります。

急増するデータ量にクラウドが追い付かない

　なんでもクラウドで扱う時代におかしな話ですね。

　例えば、デジカメで撮った写真、全部をクラウドに上げるのは大変です。4K カメラの動画は非圧縮で 1 時間 1 テラバイトとなり、アップロード時には圧縮しますが、加工前の元データは手元に残しておきませんか？クラウドに上げられないデータは活用されず、ストレージの肥やしとなります。クラウドが追い付かないデータの急増はすでに始まっているのです。

　「クラウドが追い付かない」原因は、下記の関係性です。

接続数の増加 >> ネットワーク帯域の拡大

数式による解説 $ \because \displaystyle \lim_{ x \to \infty } n \times \log n > 2^{x/1.5} \times 3 $ $$ \scriptsize{ フィボナッチ数: n = \frac{ \phi ^ x - (-\phi)^{-x} }{ \sqrt{5} } , \ 黄金比: \phi = \frac{ 1 + \sqrt{ 5 } }{2} }$$ 　左辺はデータ量、右辺はネットワーク帯域を表します。
　データ量はインターネットの 1 接続ごとの転送量で、接続数 (ノード数) を変数とするメトカーフの法則に依存します。ノード数はネズミ算で増えるため、フィボナッチ数で表現されます。ネットワーク帯域はムーアの法則に乗数をかけたギルダーの法則です。ノード数が急激すると扱うデータ量がべき乗で増えるため、ギルダーの法則が表すネットワーク帯域よりも大きくなります。

総務省 公表資料 平成27年11月分

　総務省の資料でも、接続数の増加に対してノードあたりの通信トラフィックは急増しています。これに対応するため、FTTH や 4G でネットワーク帯域は飛躍的に改善されました。「ギガ」が足りないという話も 5G で落ち着くでしょう。しかし、デジカメの画素数が増大し、IoT の時代となり、画像・動画データの急増が見込まれています。Amazon Go のような無人店舗が普及すれば、IoT のカメラは爆発的に増えるでしょう。それら IoT をクラウドで処理するためには、データをクラウドにアップロードしなくてはなりません。スーパーマーケット 1 店舗あたり、設置カメラは 100 台を超えるでしょう。1 万店舗あったら、100 万台の 4K 動画が発生します。これを本当にクラウドで処理するのでしょうか？

Edge Computing

　ネットワーク帯域不足の解決策として、Edge Computing が提唱されています。Cloud に対して Fog などとも呼ばれます。クラウドだけで処理せず、ローカル側でも処理する考え方です。プライバシー管理の容易さやレスポンスの良さもメリットです。

　問題は、管理の手間です。クラウドが広まったきっかけに、Windows Update が大変すぎるから全部クラウド側にデータを保存するようになった事があります。自社でメールサーバーを持たずすべて Gmail でまかなうのも、管理コスト削減が目的です。

DevOps

　作った人がメンテする、これが一番確実です。
　パソコンの時代、誰かが作ったアプリは使う人がメンテしていました。ある人はどこまでも自分で直せるようにオープンソースの世界へ、またある人は自動更新にお任せを、そして SIer は知らない大企業が作った OS を詳細が分からないままメンテしお客様に提供するというもっとも過酷な仕事をしていました。
　クラウドの時代になり、プログラムコードはサービス提供会社の手元に残る事になりました。ここから、開発者がそのままメンテナンスも兼任する DevOps というスタイルが生まれました。お客様にも SIer にも負担を強いない、もっとも自然な手法です。

Hybrid Computing

　Edge Computing と DevOps、双方理想的な手法ながら両立できませんでした。
　弊社では、この二つを両立させた Hybird Computing を提唱しています。Edge Computing では「データ量 >> ネットワーク帯域」の解決を、DevOps では管理コストの削減が可能です。
